The Button component is in parity with its contract. Three drift instances were found in the HITL approval flow on 2026-04-30 and resolved the same day — all hardcoded color overrides (`bg-emerald-700`, `text-red-400`, `bg-foreground text-background`) were replaced with the canonical CVA variants (`default`, `destructive`). The contract is clean.

## Drift Instances

### 1 — HitlPanel.tsx:145 · Approve button · CRITICAL

```tsx
<Button
  size="sm"
  className="flex-1 h-7 text-xs gap-1 bg-emerald-700 hover:bg-emerald-600 text-white border-0"
>
```

`bg-emerald-700` and `text-white` are hardcoded Tailwind utilities. Neither maps to a CSS custom property. The canonical equivalent is `variant="default"` which uses `--primary` / `--primary-foreground` and adapts to dark mode. The hardcoded green will not respond to theme changes and has no dark mode counterpart.

**Fix:** remove the `className` override, use `<Button size="sm" variant="default">`.

---

### 2 — HitlPanel.tsx:154 · Reject button · CRITICAL

```tsx
<Button
  size="sm"
  variant="outline"
  className="flex-1 h-7 text-xs gap-1 text-red-400 border-red-500/30 hover:bg-red-950/30 hover:text-red-300"
>
```

`text-red-400`, `border-red-500/30`, `hover:bg-red-950/30` are hardcoded. The system defines `--destructive` for exactly this purpose. The `variant="destructive"` already encodes the semantic intent — the className override duplicates it with an incompatible color.

**Fix:** `<Button size="sm" variant="destructive">` — drop the className.

---

### 3 — HitlApprovalCard.tsx:284 · Approve button · HIGH

```tsx
<Button
  size="sm"
  className="flex-1 h-7 px-3 text-[11px] font-medium rounded-md bg-foreground text-background hover:bg-foreground/90 border-0"
>
```

`bg-foreground text-background` inverts the foreground/background CSS variables manually — this is what `variant="default"` already does via `--primary` / `--primary-foreground`. The manual inversion will break in dark mode because `--foreground` is not the same as `--primary`. Additionally `h-7` (28px) overrides the `sm` size token of `h-8` (32px), creating a one-off height not present anywhere else in the system.

**Fix:** `<Button size="sm" variant="default">` — let the variant handle colors and drop the manual inversion.

## What's Consistent

The Button component itself is correct. The CVA config uses `--primary`, `--secondary`, `--destructive`, `--muted`, `--accent`, `--ring` throughout — no hardcoded values in the source. The drift is entirely at the call site: consumers bypassing the variant system with raw className overrides.
